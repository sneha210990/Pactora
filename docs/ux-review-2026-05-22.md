# Pactora UX Review — May 22, 2026

## Scope and method

This is a **heuristic UX review** based on the current Next.js UI implementation and user flows visible in code:

- Landing page (`/`)
- New review/upload entry point (`/deals/new` linkage)
- Multi-step review workflow (`/review/*`)
- Summary and action surfaces (redline/export/feedback)

Methods used:

1. Nielsen + Baymard-style heuristics (clarity, feedback, control, error handling).
2. Conversion funnel lens (visitor → first value → retained usage).
3. Accessibility pass (keyboard/screen reader/color contrast semantics visible in code).

---

## Executive summary

Pactora has a strong core proposition and a generally clean interaction model, but the UX currently over-indexes on **expert confidence** and under-delivers on **first-time guidance and trust calibration**.

Top opportunities:

1. **Shorten time-to-first-value** with a guided “first run” and expected processing time/progress confidence.
2. **Reduce cognitive load** in the summary/cards by moving from dense analysis blocks to progressive disclosure with “what should I do next?” actions.
3. **Improve trust scaffolding** by surfacing confidence, provenance, and “why this matters commercially” consistently.
4. **Accessibility + inclusivity hardening** (contrast pairs, keyboard focus visibility, status announcements).

If only 3 changes are shipped this month, do:

- Add upload+analysis onboarding checklist with live progress estimates.
- Add “Recommended next action” row to each high/medium flag card.
- Add persistent confidence/verification indicators with tooltips and glossary links.

## Journey review

## 1) Landing page (`/`)

### What works
- Clear headline and singular CTA (“Start contract review”).
- Strong visual hierarchy and low distraction.
- Beta feedback pathway is visible.

### Friction
- The page explains *what Pactora does* but not *what happens next* in concrete terms (file types, duration, output sample, security reassurance in-context).
- Social proof/trust markers are absent on the main conversion surface.
- No “preview of result quality” before commitment.

### Recommendations
- Add a compact 3-step strip under hero: “Upload contract → AI flags + clause excerpts → negotiation-ready redlines.”
- Add average processing time estimate and supported file conditions near CTA.
- Add trust microcopy (data handling, storage duration, deletion controls) inline near start action.

---

## 2) Review progress/navigation (`/review/*`)

### What works
- Stepper provides good structure and orientation.
- Current/past/future visual states are implemented.
- Horizontal overflow supports narrow viewports.

### Friction
- Progress indicates sequence but not completion criteria (e.g., reviewed vs skipped vs pending evidence).
- Checkmark state is positional (“past”) rather than semantic (“completed”).
- No explicit blocker indicators when critical fields are missing.

### Recommendations
- Introduce true completion status per step (Completed / Needs attention / Not started).
- Add optional “jump to highest risk” quick-nav chip.
- Show missing inputs inline in step labels (e.g., “Liability cap — ACV needed”).

---

## 3) Clause flag card UX (summary + detail surfaces)

### What works
- Expand/collapse pattern is familiar.
- Risk badges and page references are useful for legal workflows.
- Negotiation ladder + alternative generation align with user jobs-to-be-done.

### Friction
- High information density on expansion can overwhelm non-legal users.
- “Unverified” indicator appears conditionally and may be missed; trust cues are not standardized.
- The first action is not always obvious (“what should I do now?”).

### Recommendations
- Reorder card content to: **Risk → Why it matters → Recommended action → Evidence → Negotiation script**.
- Add fixed action buttons for each flag: “Accept suggestion”, “Mark for legal”, “Dismiss with reason.”
- Normalize trust panel with confidence level + extraction/source context on every card.

---

## 4) Summary decisioning (“Should we sign?”)

### What works
- Clear top-line verdict language (review required / proceed with caution / acceptable risk).
- Composite risk score helps triage quickly.

### Friction
- Scores can imply false precision without methodology/context.
- Risk thresholds are hidden from user mental model.
- “Proceed with caution” lacks concrete follow-up checklist.

### Recommendations
- Add “How this score is calculated” explainer drawer.
- Convert verdict into task-based checklist with owners: Procurement / Legal / Security.
- Show scenario delta: “If you resolve top 2 flags, score drops from X to Y.”

---

## 5) Accessibility and inclusive UX

### Strengths
- Semantic navigation landmarks exist in key components.
- `aria-current` on progress step is good.

### Risks / gaps
- Dark UI with muted text tones risks contrast failures on some displays.
- No visible sign in this pass of ARIA live announcements for async operations (redline generation/loading/errors).
- Dense cards may be hard for keyboard-only users if focus styling/order is subtle.

### Recommendations
- Run automated a11y checks (axe/Pa11y) in CI for key flows.
- Add polite/live region announcements for async status changes.
- Ensure robust visible focus ring tokens across all interactive elements.

## Suggested UX success metrics

- **Activation**: % of new users reaching first completed summary.
- **Time to first value**: median time upload → first actionable flag.
- **Actionability**: % of high-risk flags with user action taken.
- **Trust calibration**: % of users opening confidence/provenance details.
- **Outcome proxy**: % sessions exporting PDF/redlines after review.
