# Pactora Integrity Engine

The Pactora Integrity Engine is a deterministic legal document integrity validator for contract sets. It is intentionally not a chatbot workflow: every issue is generated by an isolated validator over normalized contract structure, definitions, references, and locations.

## Folder structure

- `lib/integrity-engine/types.ts` — normalized contract, location, issue, validator, and report contracts.
- `lib/integrity-engine/parser.ts` — text-to-structure parser that produces sections, definitions, references, term usages, lowercase term usages, and structural targets.
- `lib/integrity-engine/definition-extractor.ts` — extensible legal drafting definition patterns such as `"Affiliate" means` and `"Services" shall mean`.
- `lib/integrity-engine/reference-graph.ts` — reference extraction and graph target normalization for sections, schedules, annexes, exhibits, attachments, and related documents.
- `lib/integrity-engine/validators/*` — composable deterministic validators.
- `lib/integrity-engine/engine.ts` — report orchestration and validator execution.
- `app/api/contracts/integrity/route.ts` — MVP API route for JSON text or multipart contract uploads.
- `prisma/schema.prisma` — suggested PostgreSQL persistence model for parsed structures and audit-grade run history.

## API examples

### JSON contract set

```http
POST /api/contracts/integrity
Content-Type: application/json
```

```json
{
  "documents": [
    {
      "id": "msa-1",
      "title": "Acme MSA",
      "kind": "msa",
      "text": "1. Definitions\n\"Affiliate\" means any entity...\n2. Services\nCustomer will receive the services described in Schedule 3."
    },
    {
      "id": "sow-1",
      "title": "Statement of Work 1",
      "kind": "sow",
      "text": "1. Scope\nSupplier shall provide Services under the MSA."
    }
  ]
}
```

### Multipart uploads

Upload one or more files under `contracts`. The route supports PDF, DOCX, and best-effort legacy DOC through the existing document extraction layer.

```bash
curl -X POST http://localhost:3000/api/contracts/integrity \
  -F "contracts=@msa.docx" \
  -F "contracts=@sow.pdf" \
  -F "kind_0=msa" \
  -F "kind_1=sow"
```

## Deterministic validator behavior

1. **Undefined Defined Terms** — flags repeated capitalized legal terms with no matching definition anywhere in the uploaded document set.
2. **Duplicate Definitions** — flags same normalized term defined more than once; severity increases when definition text differs.
3. **Dead Definitions** — flags definitions with no later usage outside the definition line.
4. **Broken Cross References** — flags references to missing sections, schedules, annexes, exhibits, attachments, and documents.
5. **Inconsistent Capitalization** — flags lowercase usage only where a defined term exists and the lowercase usage appears in drafting contexts likely to intend the defined term.
6. **Multi-Document Integrity** — builds one structural target set across all uploaded documents, so an MSA can validate schedules, annexes, SOWs, DPAs, and order forms uploaded separately.

## Edge cases handled in the MVP

- Definitions can appear in ordinary prose or definition-list drafting.
- Duplicate definitions are checked across the complete document set, not just within one file.
- Section references validate against parsed clauses in any uploaded document.
- Schedule, annex, exhibit, and attachment headings create structural targets even when they are separate uploaded files.
- Common legal role terms such as `Customer` and `Supplier` are treated conservatively when seen once without a definition to reduce false positives.
- Lowercase capitalization issues are not raised unless the uppercase defined term exists.

## Suggested unit tests

- `parseContract` should nest `1.1` under `1` and preserve line ranges.
- `extractDefinitions` should extract `"Affiliate" means`, `"Services" shall mean`, and definition-list entries.
- `extractReferences` should normalize `Section 9.2`, `Schedule 3`, `Annex B`, and `DPA` targets.
- `duplicateDefinitionsValidator` should flag different definitions of the same term across an MSA and schedule.
- `brokenReferencesValidator` should pass when `Schedule 3` is uploaded separately and fail when absent.
- `deadDefinitionsValidator` should ignore a term used later in another related document.
- `inconsistentCapitalizationValidator` should flag `the services` only when `Services` is defined.

## Future AI enhancement points

AI should augment, not replace, deterministic validation. Safe insertion points include:

- Clause-boundary enrichment when DOCX/PDF layout parsing is ambiguous.
- Semantic clause drift detection across related agreements after deterministic section alignment.
- Playbook deviation explanations that cite deterministic issue IDs and exact source locations.
- Fallback clause comparison and negotiation intelligence layered over stored parsed structures.
- Obligation extraction and risk scoring that reference persisted `ContractSection` and `IntegrityIssue` records for auditability.
